# 設計判断

*[English](./decisions.md) | 日本語*

このドキュメントは kanabun を形づくる選択と ── 同じくらい重要な ── 検討して見送った
代替案を記録します。初期のプロダクト/設計インタビューの結論をまとめたものです。

## プロダクトの位置づけ

- **動機:** *実用ツール*。学習だけでなく、実プロジェクトで使えるものを目指す。
- **成功の定義(約1年):** kanabun 上で TodoMVC が動き、他者が試せる程度にドキュメント
  が揃っている状態。
- **最初のドッグフーディング対象:** 設計検証は TodoMVC 風のデモで十分。具体的な実アプリ
  は後で。
- **直近のスコープ:** 横展開の前に、まず**コアを鉄壁にする**(Phase 1)。

## 技術的な中核判断

### 1. 仮想 DOM ではなく signals でリアクティブに

変わった所だけ更新する。仮想 DOM も差分計算も実装しない。コアは DOM なしで完全に
ユニットテストできる、純粋なリアクティブグラフ。

### 2. 明示 getter(Solid 流)── ゆえに**コンパイラなし**

リアクティビティの表記は `const count = signal(0)`、読みは `count()`、書きは
`count.set(v)`。

これは最も影響の大きい判断でした。元の要求にあった次の2つが両立しないからです。

- *Svelte 風の手触り*(`let count = $state(0); count++`)
- *コンパイラなし*

この2つは共存できません。`count++` のような素のローカル変数代入を実行時に検知するのは
不可能で、Svelte 5 の `$state` が魔法に見えるのは**コンパイラ**がそのコードを追跡可能な
`get`/`set` 呼び出しに書き換えているからです。つまり「Svelte の手触り」は本質的に
コンパイラの産物です。

検討した3つの解決策:

| 案 | 手触り | コンパイラ | 結論 |
| --- | --- | --- | --- |
| **A. 明示 getter(Solid)** | `count()` / `count.set()` | 不要 | **採用** |
| B. Proxy オブジェクト(Vue) | `s.count` / `s.count++` | 不要 | 却下: Proxy のエッジケース(分割代入で購読が切れる、ネスト/配列/Map 対応)が「鉄壁」を難しくし、テンプレート境界では結局 getter が要る。 |
| C. Svelte 風(`count++`) | 最良 | **必須** | 当面却下: コアを固める**前に**コンパイラという大物を背負う ── 選んだ優先度の逆。 |

**A** を採用した理由は、他のすべての回答(*コンパイラなし*・*コア優先*・*集中投下*)と
一直線に整合する唯一の案であり、SolidJS が既に実証済みの戦略だからです。`()` は小さく
正直なコストで、ソース上に見える**購読そのもの**です。

これは C を永久に閉ざすものではありません。将来コンパイラが同じランタイムの上に
`$state` 糖衣を載せることは可能です。A は床であって天井ではありません。

### 3. テンプレートは JSX(後のフェーズ)

テンプレートを導入する際は JSX/TSX にします。TypeScript が要素・属性・props・埋め込み式を
ネイティブに型チェックし、エディタの補完やハイライトも無償で効きます。kanabun が用意する
のは JSX 型定義と小さな `jsx`/`jsxs`/`Fragment` ランタイムだけ ── **LSP も独自 DSL も
作りません**。(tsconfig は既に `jsxImportSource: "@kanabun/core"` に配線済み。)

### 4. ツールには Bun、ただしコアはランタイム非依存

`@kanabun/core` は標準 JS / Web API のみを使うので、どこでも動きます。Bun は開発体験の
ため(テストランナー、将来はバンドラ/dev サーバー)だけに使い、コアを汚さない薄い CLI/dev
層に閉じ込めます。

### 5. 依存ゼロ ── ランタイム*かつ*開発環境

「依存ゼロ」は出荷バンドルだけでなく開発環境にも及びます。

- **ランタイム:** ゼロ。コアは何も import しない。
- **開発:** 型専用の依存 `@types/bun` を1つだけ、意図的な例外として許可
  (`bun:test` / Bun の型を提供する)。文字通りのゼロを目指して手書きの ambient シムを
  試作しましたが、`@types/bun` を選びました。新しいテストマッチャーを使うたびに拡張が
  必要なシムより、正確で保守された型のほうが優れます。
- **TypeScript** は本プロジェクトが許可するツール(憲章「Bun と TypeScript のみ」)であり、
  `package.json` にベンダリングせず `bunx tsc` でオンデマンド取得します。
- **CI 基盤**(`actions/checkout`、`oven-sh/setup-bun` などの GitHub Actions)は
  プロジェクトの依存グラフに含まれず、この規則の対象外です。

結果として `bun install` が入れるのは `@types/bun` のみ、ブラウザに届くのは標準 JS のみ。

## 規約とツール

- **テストファイル名**は `*.spec.ts`。
- **パッケージマネージャ/ランナー**は Bun。`bun test`、`bun run typecheck`
  (`bunx tsc --noEmit`)を使い、npm/yarn 相当は使わない。
- **カバレッジ**は Bun 内蔵の `--coverage`(text + lcov)で計測し、`bunfig.toml` に
  閾値 0.9 を設定。現状コアは 100%。
- **CI**(`.github/workflows/ci.yml`)は push / PR ごとに型チェック・テスト・カバレッジを
  実行。
- **依存の追加**は明示的に正当化すべき危険信号。`skeptical-reviewer` エージェント
  (`.claude/agents/`)がタスク完了報告の前にこれをチェックします。

## リアクティブコアの意味論(Phase 1)

伝播アルゴリズムは push–pull の「カラーリング」方式(Reactively/Solid 流)。後から変える
のが最もつらい次の2つの意味論を、ここで意図的に決め切りました。

- **glitch-free:** ダイヤモンド(`a → b, a → c, (b,c) → d`)で `a` への1回の書き込みが
  各ノードを高々1回しか再計算せず、不整合な中間状態(`d` の一時的な誤値)を見せない。
  テストで検証済み。
- **自動購読解除:** 依存は毎回採り直すので、あるソースを読まなくなった計算はそのソースから
  購読解除される(リークなし、動的依存も自然に動く)。テストで検証済み。

破棄はリーフで明示的に行います。`effect` は disposer を返し、`onCleanup` / 返り値の
クリーンアップが後始末を担います。

所有権は(さらに先送りせず)Phase 2 で必要になった時点で追加しました。
`createRoot(fn => …)` がスコープを作り、その中で生成された計算はまとめて破棄され、
再実行する effect は前回の実行で作った子を破棄します(リークなし)。これは Phase 1 の
伝播 ── とそのテスト ── に手を入れない**追加レイヤー**として実装しました。

## 描画の判断(Phase 2)

### ランタイム JSX、コンパイラなし

`jsx`/`jsxs`/`Fragment` は呼ばれた時点で**実 DOM を即時生成**します
(hyperscript + signals 方式)。仮想 DOM も差分計算も独自コンパイラもありません。
これは「ランタイムのみ」の判断を守りつつ、JSX の型チェックとエディタ支援を無償で得ます
(TypeScript は `jsxImportSource` 経由で我々の `JSX` 名前空間から型を解決)。
コンポーネントは**一度だけ**実行され(Solid モデル)、再実行するのはリアクティブな式だけです。

### 反応式の規約:関数がリアクティブ

コンパイラがないので、何がリアクティブかをソースが示す必要があります。規則は
**子・属性の値が関数ならリアクティブ**(`effect` でラップし、コメントマーカーで位置を
固定)、それ以外は一度だけ設定。`on*` は常にイベント。よって `{count}` や
`{() => count() * 2}` はリアクティブ、`{count()}` は一度読むだけ。これがコンパイラ
なし選択のための唯一の儀式で、小さく明示的です。

### DOM 依存なしで DOM をテストする

レンダラは DOM を必要としますが、Bun は DOM を持たず、jsdom / happy-dom は依存ゼロの
方針に反します。そこでレンダラは `globalThis.document` を遅延解決し、テストはリポジトリ内の
小さな DOM モック(`packages/core/src/dom-mock.ts`)を差し込みます。import 時に `document` を要求しない
ので、コアはどこでも読み込めます。さらに例は CI で `bun build` によりビルドし、実際の
JSX トランスフォームをエンドツーエンドで検証します。

## 制御構文の判断(Phase 3)

### keyed リストを2層で

最難関 ── 配列の効率的な更新 ── を、独立してテスト可能な2つの小さな部品に分けました。

- `mapArray`(`<For>` 内)はアイテムごとに**ノード identity を安定化**します。参照で
  キーづけし、アイテムにつきノードを一度だけ生成、並べ替え/挿入で再利用、各アイテムは
  独自の `createRoot` で動くので、削除されたアイテムのリアクティビティは破棄されます。
- `reconcileNodes`(DOM ランタイム)は identity を基準に DOM を目標ノード列へ同期し、
  消えたものを削除、本当に位置がずれたノードだけを移動します(後ろから前へアンカリング)。

組み合わせることで、全再構築なしの keyed 更新になります。アルゴリズムは LIS 最適な最小移動
(Solid の udomdiff)ではありませんが、O(n) で正しく、TodoMVC には十分で、同じ継ぎ目の裏で
後から差し替えられます。

### `<Show>` / `<For>` は魔法ではなくコンポーネント

これらは一度実行されリアクティブな thunk を返す普通のコンポーネントで、他のリアクティブな子と
**まったく同じ挿入経路**に乗ります(レンダラ側に特別扱いなし)。`<Show>` は条件を boolean に
メモ化するので、真値の間で条件が変わっただけでは子を入れ替えません。

`<Show>` の破棄はコンパイラではなく「関数は遅延」規約に従います。素の要素の子
(`<Show><Child/></Show>`)は一度だけ生成され、非表示の間は**切り離されるだけ**でリアクティビティ
は生存します。一方、関数の子(`<Show>{() => <Child/>}</Show>`)は遅延生成されるので、非表示で
子のスコープが(owner ツリー経由で)破棄され、表示で再生成されます。両挙動ともテストで固定して
あり、トレードオフは偶然ではなく明示的です。

## スコープド CSS(Phase 4)

スコープ付きスタイルも創設時の制約 ── **ランタイムのみ・コンパイラなし・依存ゼロ**
── に収める必要があり、Svelte のビルド時セレクタ書き換えは選べません。ランタイムで
取りうる3つの形を比較しました。

| 案 | 形 | 判定 |
| --- | --- | --- |
| **A. Emotion 風 `css\`…\`** | 本体をクラスにハッシュし、ルールをその配下にスコープ、クラス名を返す | **採用** |
| B. CSS Modules 風 `css({ name: rules })` | 宣言のオブジェクト → ハッシュ済みクラス名のマップ | 却下:解析ゼロで最も堅牢だが、擬似クラス・ネスト・メディアクエリが窮屈。 |
| C. Svelte 風 属性スコープ | 通常セレクタ + 要素への `data-` 属性注入 | 却下:本格的な CSS パーサ **と** JSX ツリーへの属性注入が必要で重く、「薄く rock-solid なランタイム」という優先順位に反する。 |

**A** を選んだのは、roadmap の記述(「class をハッシュして `<style>` を注入」)と
フレームワークの哲学に直結するからです。内容ハッシュが一意なので異なるスタイル
ブロックは原理的に衝突せず、**ライブ DOM に対するセレクタ書き換えもビルドステップも
不要** ── 文字列のスコープ付けだけで済みます。`class` はただの文字列のままなので
JSX ランタイムの改修は不要で、リアクティブな切り替えは既存の「関数はリアクティブ」
規約を再利用できます(`class={() => on() ? a : b}`)。加えて、人気の *styled-components*
系エディタ支援が(`css` というタグ名に反応して)無設定で効くのも A だけで、しかも
それはエディタ拡張でありプロジェクト依存ではないため、依存ゼロ規約は無傷です。

スコープ変換は **意図的に範囲を限定**しています(100% でテスト可能に保つため)。
トップレベル宣言は `.k-hash` 配下へ。ネストブロックは `&` をそのクラスに置換、`&`
が無ければ子孫に。カンマ区切りは各部分をスコープ(`()`/`[]` 内のカンマは保持)。
条件付きグループ at-rule(`@media`/`@supports`/`@container`/`@document`/`@layer`)は
再帰して内側を再スコープし、その他の at-rule(`@keyframes`・`@font-face`)は本質的に
グローバルなのでそのまま通します。唯一の既知の制限は、波括弧のマッチが字句的なこと
── 文字列やコメント中のリテラル `{`/`}` は解釈しません(コンポーネントのスタイルでは
ほぼ不要で、残りはグローバルなスタイルシートでカバー)。dedupe はハッシュで、ライブ
`<head>` を見て判定するので、モジュールレベルのキャッシュをリセットせずとも描画を
跨いで効きます。

## コンテキスト(Phase 4)

`createContext` / `useContext` には roadmap で明記していた設計上の分岐が1つあった:
**ランタイム・コンパイラなし**のモデルでは JSX の子は*即時*評価される(コンストラクタ
引数なので)ため、`<Provider>` は子が読む前に値を設定できない。3案を比較した:

| 案 | 形 | 判定 |
| --- | --- | --- |
| **A. 関数の子** | `<Ctx.Provider value={v}>{() => <App/>}</Ctx.Provider>` — thunk は値設定の*後*に走る | **採用** |
| B. コンパイラ導入 | 子を自動で遅延させるよう JSX を書き換える | 却下:「コンパイラなし」という創設制約に反する。 |
| C. 先送り | context なしで出す | 却下:Phase 4 最後の未対応項目だった。 |

**A** を採る理由は、フレームワークに既にある規約 ──「**関数は遅延**」、まさに
`<Show>`/`<For>` が子を扱うのと同じ ── を再利用できるから。新しい考え方も、
コンパイラも、依存も不要。代償は明示的:素の(即時)子は*デフォルト*しか見えない。
この制約は黙って退行しないようテストで固定している。

実装は既存の **owner ツリー**(Solid 流)に乗る。各リアクティブノードに親リンク
(`owner`)と任意の `context` マップを追加し、`useContext` は現在の owner から上へ辿って
最も近い提供値(なければデフォルト)を返す。Provider は値を保持する owner スコープを
1つ作り、外側 owner の所有とすることで一緒に dispose される。コードを形作った要点は2つ:

- **遅延 thunk。** `<For>` のようなコンポーネントの子は*thunk* を返し、それは外側の
  `insert` が後で、Provider スコープの*外側*で作った effect の中で実行する。そこで
  Provider は、返ってきた thunk を**呼ばれるたびに owner スコープへ再突入する**よう
  ラップする(所有のみ。トラッキングは触らないので `<For>` は依存を保つ)。これが無いと
  行は値を含まない owner チェーンを辿ってしまう。直接 DOM の子はラップ不要 ── その
  バインドの effect はスコープが有効なうちに同期的に作られるため。
- **`createRoot` の親リンク。** `createRoot` は親 owner を記録するようになった
  (`owner.owner = prevOwner`)。これにより、自前の root で走る `<For>` 行からも
  リストより上の Provider を解決できる。dispose 挙動は変えない(root は依然として明示的な
  境界)。owner ツリーを上方向に辿れるようにするだけ。

## CLI の判断(Phase 5)

`@kanabun/cli` は**唯一**の Bun 依存層で、コアは Bun/Node API を一切 import しません。
`kanabun` コマンドは3つのサブコマンドを持ちます。

- `build` は `Bun.build({ target: "browser" })` のラッパー(esbuild/Vite なし)。例外を
  投げず、解決不能なエントリやコンパイルエラーは `{ success: false, logs }` として返します。
- `dev` は Bun 内蔵の `Bun.serve` 上の薄いサーバー。HTML エントリを配信(ライブリロード
  スニペットを注入)し、TS/TSX をオンデマンドでバンドルし、ファイル変更時に WebSocket で
  更新を送ります ── `.css` の編集はターゲット指定の **CSS ホットスワップ**、それ以外は
  **全リロード**(下記 [CSS HMR](#css-hmrphase-6) 参照)。リクエストハンドラ
  (`createDevHandler`)を分離し、ソケットを立てずにユニットテストできるようにしています。
- `create` は埋め込みテンプレートから動くプロジェクトを生成(ネットワーク不要)。

CLI もコアと同じ基準(カバレッジ 100%)を満たし、実 dev サーバーのテスト(エフェメラル
ポートで起動 → HTML + バンドル JS を取得 → ファイル変更で WebSocket リロード発火を検証)も
含みます。

## ルーター(Phase 6)

`@kanabun/router` は history ベースの別パッケージです。コアと同じ創設制約 ──
**依存ゼロ・コンパイラなし・ランタイム非依存** ── を守り、新しい仕組みは一切持ち込みません。
既存の signals と owner ツリー context、そして `<Show>`/`<For>` の「関数は遅延」規約に
そのまま乗っています。

- **history ソースの抽象化。** ルーターの状態と「URL が実際にどこにあるか」の間に
  `RouterSource` という薄い継ぎ目を置きました。`createBrowserSource` は `window.history`/
  `location`/`popstate` を駆動し(`window` は **遅延解決** ── import 時に DOM を要求しない)、
  `createHashSource` はルートを URL ハッシュ(`#/path`)に置き、書き換えルール無しの静的ホスト
  (GitHub Pages・S3・ファイルサーバー)でも直リンク・リロードが動くようにします。
  `createMemorySource` はインプロセス実装でテストや非ブラウザ(SSR)用です。この継ぎ目が、
  jsdom 無しでの 100% ユニットテストを可能にし、**かつ**ハッシュルーティングを書き直しでなく
  約 20 行の追加で済ませました(`<Router>` は無変更)。
- **リアクティブな現在地。** `<Router>` は現在パスを保持する signal を1本持ち、ソースを
  購読し(`onCleanup` で解除)、`computed` で `RouterLocation`(`pathname`/`search`/`hash`/
  `query`)に解析します。`push`/`replace` は通知せず、ナビゲート後に signal を同期更新します
  (back/forward だけが購読を発火)。
- **`<Route>` は `<Show>` と同じ意味論。** マッチ結果を **boolean にメモ化**するので、
  マッチ中は内容を一度だけ構築し、非マッチで破棄します。一方 params は下層で
  リアクティブに更新され続けます(同じルート内のパラメータ変化では再構築しない)。
  マッチャ(`matchPath`)は静的 / `:param` / 末尾 `*wildcard` を扱う純関数です。
- **排他マッチング(と 404)の `<Routes>`。** 単独の `<Route>` は独立トグル(マッチした
  ものは全部描画)なので、キャッチオールには不適。そこで `<Routes>` は **最初にマッチした
  子1つ**を描画し、どれもマッチしなければ共有の `fallback` を出します。コンパイラなし・
  JSX 即時評価のもとでの工夫は、`<Route>` が **マッチ状態を属性として持つ thunk**
  (`$matched` / `$content` ── 関数はプロパティを持てる)を返すこと。これにより単独でも
  描画でき、`<Routes>` はその属性を読んで1つを選ぶ(Solid の `<Switch>`/`<Match>` に相当)。
  名前は Solid の `<Switch>` ではなく React Router 流の `<Routes fallback>` を採用。
- **破棄は明示的な `createRoot` スロットで。** 遅延読みがルーターを解決できるようにする
  context ラップは、ルートのサブツリー全体を **安定した** Router owner の下で走らせるため、
  再実行する insert effect の `disposeOwned` が前のルートに届かず、切替のたびにリークします。
  そこで `<Route>`/`<Routes>` は内容を小さな「使い捨てスロット」(専用 `createRoot` で所有)
  経由で描画し、切替/アンマウント時に前を破棄します ── `<For>`/`mapArray` と同じ明示破棄の型。
- **params の受け渡しは2系統。** `component`/関数の子に accessor を直接渡しつつ、
  `RouteContext` 経由で `useParams()` でも読めます。後者は core の context を流用するため、
  子孫は **関数の子**(遅延)配下にある必要があります(即時の子はデフォルト=空を見る、
  という既存の制約と同じ)。
- **`<Link>`。** クライアント遷移する `<a>`。素の左クリックだけ横取りし、修飾キー付き
  クリック・左以外のボタン・`_self` 以外の `target`・外部/`mailto:` リンクはブラウザ既定に
  委ねます。

ルーターもコアと同じ基準で、全ソース カバレッジ 100%・`tsc` クリーンを満たします。
`examples/router` は実ブラウザ(`snapshot` スキル)で遷移(リロードなし)・`:id` 解決・
`useLocation` 表示・スコープド CSS を確認しています。(コミット済み VRT ベースラインの
追加は follow-up ── pinned Playwright コンテナで撮る必要があるため。)

### ネストルーティング(Phase 6)

レイアウト + 子ルートを、やはり **新しい仕組みなし**で。相対マッチングはフラット版と同じ
owner ツリーの context に乗ります。

- **末尾ワイルドカードがレイアウトを作る。** マッチャ `matchRoute`(`matchPath` と並んで
  エクスポート)は params に加えて `rest` を返します ── `*` ワイルドカードパターン
  (`/users/*`)では **プレフィックス** マッチの余り(ネスト側が自分で params をデコードできる
  よう *raw* のまま)、完全一致なら `null`。`<Route>` は「直近のマッチした祖先が残した余り」
  (新しい `RelPathContext`、既定=フルパス名)に対してマッチするので、パターンはネスト位置に
  **相対**で書けます。
- **ネストしたルーター自体が outlet。** 子ルート設定を流し込む `<Outlet>` プレースホルダ
  (親がマッチする前に子 `<Route>` を構築する必要がある=eager-children 問題)ではなく、
  レイアウトは **自身のコンポーネント本体内**でネストした `<Routes>`/`<Route>` を描画します。
  これは親ルートの context 配下で走るので余りパスを見られます。「コンポーネントは描画位置に
  置かれた関数」という kanabun のモデルと整合し、追加コンポーネントも不要。
- **params は連鎖でマージ。** ネストした `<Route>` は自分の捕捉を祖先に重ねる
  (`{ ...parent, ...local }`)ので、子孫の `useParams()` は連鎖全体(`{ org, id }`)を読めます。
  トップレベルでは未マッチ読み取り向けの安定した空参照を維持します。
- **注意点は1つ、コアと同じもの。** ネストした `<Routes>` はホスト要素
  (レイアウトの chrome ── `<div class="users-layout">`)の内側に置く必要があります。自前の
  リアクティブ insert 境界を得るためです。親ルートへ素の thunk をそのまま返すと親のトラッキングに
  平坦化され(`reconcile` に届いた関数は一度だけ読まれる)、内側の遷移ごとにレイアウトが
  再構築されます。フラットの例も既にこれに従っています(`<Routes>` は `<div class={shell}>` の
  内側)。レイアウトは元々ラッパーを持つので追加コストはありません。

`examples/router` がこれを示します:`/users/*` レイアウトがマスター一覧を常駐させつつ、
ネストした `<Routes>` が詳細ペインを差し替えます。*相対 `<Link>` href(現在のルート基準で
解決)は follow-up ── href は絶対指定です。*

## エラーバウンダリ(Phase 6)

`catchError`(コアのプリミティブ)と `<ErrorBoundary>`(コンポーネント)により、
サブツリーがアプリ全体を巻き込まずに失敗できます。context と同様、新しい仕組みは
持ち込まず、既存の **owner ツリー** に乗っています。

- **エラーハンドラは owner ツリー context そのもの。** `catchError(tryFn, handler)`
  は、private な `ERROR` シンボルの下に `handler` を持つ owner スコープを作ります ──
  `createContext` の `Provider` が値を格納するのと全く同じやり方です。計算が throw すると、
  伝播コアは **owner チェーンを上に辿って** 最も近いハンドラを探します(`handleError`)。
  `useContext` が値を上に辿るのと対称です。ハンドラが見つからなければ再 throw されるので、
  ガードされていない失敗は握り潰されず、ちゃんとホストまで届きます。
- **コア側の捕捉点は1か所。** 派生計算の `update()` が `fn()` を 1 つの `try`/`catch` で
  囲みます。throw 時は収集途中の依存を捨て、自身を CLEAN にして(再検証 pull がループ
  しないように)エラーをルーティングします。これがリアクティブコアへの *唯一* の変更で、
  正常系では no-op ── glitch-free な伝播には手を付けていません。
- **生成時・更新時の両方を捕捉。** 子孫の effect/computed が *再実行* 中に throw した場合は
  その `update()` で捕捉してルーティング。子の *生成* 中(即 throw するコンポーネント関数等)
  に throw した場合は `catchError` 自身の `try`/`catch` が同期的に捕捉します。ガード対象の
  スコープは owner ツリーに登録されているので、その配下で作られた effect が *後の* flush で
  初めて走って throw しても、ちゃんとハンドラに届きます。
- **`<ErrorBoundary>` は子を一度だけ構築し、あとは *選ぶ* だけ。** コンポーネントはガード対象の
  子を **専用の `createRoot` で先に構築** し、`catchError`(ハンドラが `failure` signal を set)で
  包みます。レンダー thunk は `failure` を read して、構築済みの子か fallback を返すだけ ──
  毎回の再構築は **しません**。この分離が *ネスト* を安全にします:ネストした境界のリアクティブ
  read は親のスロット内で走るため、もし子を thunk 内で再構築していたら、内側の失敗が親を再実行し、
  内側境界を作り直し(捕捉済みのエラーが消える)、永遠に再 throw してしまいます。一度だけ構築する
  ことでこのループを断ちます。`reset` は再構築します(先に古い・壊れた `createRoot` サブツリーを
  破棄)。再構築は `batch` で包み、破棄・エラークリア・再構築が境界の再レンダー前に確定するように
  しています。`fallback` は静的ノードか `(err, reset) => node` のどちらかで、コンパイラなしで
  React 相当の使い勝手にしています。
- **子は関数(遅延)── `<Show>`/context と同じ。** 子を thunk で包めば *生成* もガードされます。
  素の(即時)子は境界より前に作られるので、後の更新しか捕捉できません。

これも同じ基準 ── 全ソース カバレッジ 100%・`tsc` クリーン・依存ゼロ・ランタイム非依存。

## 開発時警告(Phase 6)

**コンパイラがない**ため、ビルド時には捕まえられない種類のミスがあります。その受け皿
として `setDev(true)` で有効化する **ランタイム診断** を少数用意しました ── 「関数=
リアクティブ / getter は明示」という規約が間違えやすくするポイントを狙います。

- **既定オフ・オプトイン。** 本番もテストスイートも、明示的に有効化しない限り無音
  (コンソールを汚さない)。オフのときは `warn` が即 return するのでホットパスのコストも
  ゼロ。`kanabun dev` は、(defer される)アプリのモジュールより前に classic な inline
  `<script>globalThis.__KANABUN_DEV__ = true</script>` を注入して自動で有効化します ──
  配信ページとバンドルされた core は `globalThis` を共有するので、バンドル間の import
  配線は不要です。他の環境では `setDev(true)` で強制オンにできます。
- **重複排除と差し替え可能なシンク。** 同一メッセージは最大1回だけ出力(再実行する
  エフェクトがコンソールを溢れさせない)。警告は `setWarnHandler` 経由で流れるので将来の
  dev オーバーレイが横取りでき、`null` で `console.warn` に戻ります。触れるのは
  `console`/`globalThis` のみなのでモジュールはランタイム非依存のまま。
- **何を検知するか**(いずれもコアが既に持つ状態から検知でき、誤検知が少ないもの):
  - **owner 外で作られた `effect()`** ── 自動 dispose されない(`render`/`createRoot` の
    owner が無い)。リークの恐れ。`computed` は対象外 ── モジュールレベルの派生値は正当で
    副作用も無いため。
  - **owner 外の `onCleanup()`** ── コールバックが黙って一度も実行されない。
  - **owner 外の `onMount()`** ── 実行はされるがライフサイクルに紐づかない(中の
    `onCleanup` は無視される)。
  - **computed の評価中にシグナルへ書き込み** ── 純粋であるべきものの中の副作用(走行中の
    `listener` が非エフェクトの派生であることで検知)。エフェクトは書き込み可なので対象外。
- **なぜ「`{count}` のつもりで `{count()}` と書いた」を検知しないのか。** それが動機となる
  例ですが、ランタイムでは確実には検知できません ── JSX の子が作られる頃には `count()` は
  既にただの値に潰れていてリテラルと見分けがつかず、同期レンダー中のシグナル読み取りは
  正当なことも多いからです。確実に捕まえるにはコンパイラが要りますが、それは founding
  constraints が禁じています。上記の警告は、推測せずに反応グラフから *検知できる* ものに
  絞っています。

これも同じ基準 ── 全ソース カバレッジ 100%・`tsc` クリーン・依存ゼロ・ランタイム非依存。

## SSR・ハイドレーション・SSG(Phase 6)

設計の根っこの判断:**SSG は独立した機能ではなく、SSR をリクエスト時ではなくビルド時に
走らせたもの**。サーバ側は1つのプリミティブ(`renderToString`)、クライアント側も1つ
(`hydrate`)を共有し、違いは *`renderToString` をいつ走らせるか* と *出力をディスクに
焼くか* だけ。SSG を別物として設計すると描画経路が二重になる。だからこうする:

```
core:  renderToString(() => <App/>)  → HTML 文字列、DOM 不要、シグナルは1回読み
       hydrate(() => <App/>, root)   → サーバマークアップ上にライブツリーをマウント
                                       (クリア→再描画。ノード引き取りはしない)
SSR  = リクエスト時に renderToString し、レスポンスで返す
SSG  = ビルド時に renderToString し、.html に書き出す(+任意で hydrate)
```

よって実装順は **SSR/ハイドレーションが先**。SSG はその後、薄い CLI の prerender ループ
に縮み、**新しい描画ロジックは不要**。

- **`renderToString` は core に置き、`document` に一切触れない。** 実 DOM 向けの
  `render`(`dom.ts`)は動的な箇所を `effect` で包んで実 DOM を歩く。サーバには DOM が無く
  反応性も要らない(マークアップは一度作るだけ)ので、`renderToString` は別の走査として
  **リアクティブ値をちょうど1回読む**(accessor を呼ぶだけで購読しない)で文字列を連結する。
  純粋な標準 JS(`Bun.*`/`fs`/`document` 不要)なので `packages/core` に置けてランタイム
  非依存の規約を守れる。SSG のファイル書き出しだけが `fs` を要し、それは CLI/Bun 層の仕事。
- **ハイドレーションはノード単位の「引き取り」ではなく「マークアップ上へのマウント」── これは
  怠慢ではなく founding 制約の帰結。** サーバ HTML が先に描画され(高速初回描画・SEO)、その後
  `hydrate` がそれをクリアして同じ位置にライブな反応ツリーをマウントする。バイト列が同一なので
  視覚的なちらつきは無い。やら *ない* のは既存サーバノードの引き取り(その場で再利用しリスナ/
  effect だけ接続)。理由:JSX ランタイムは DOM を **eager かつ bottom-up** に組む ──
  `jsx("div", {children:[jsx("span", …)]})` は外側より先に内側の `jsx`(と `createElement`)を
  評価するので、子ノードはサーバツリーのどこに属すか分かる前に作られてしまう。ノード単位の
  引き取りにはサーバ DOM を上から辿るカーソルが要り、それにはコンパイラが吐くハイドレーション
  マーカーかテンプレート複製(Solid のやり方)が要る。どちらも「コンパイラなし」制約で除外される
  ので、引き取りは v1 の約束ではなく **将来作業** として記録する。受け入れるコストはノードの
  その場再利用(とマウント部分木の一時的なフォーカス状態)の喪失 ── 正しさや初回描画ではない。
- **スコープド CSS はサーバの `<head>` に収集する ── これが共通の落とし穴。** `css` の
  `inject` は `<style data-k>` を `document.head` に足す。サーバでは設置した
  `ServerDocument` が `<head>` を *持つ* ので、render **中** に出たスタイルはそこにタダで
  集まり、`renderToString` が `head` 文字列として返す。厄介なのは import 時のスタイル:
  モジュールトップレベルの `` css`…` `` は `renderToString` が document を設置する前 ──
  `document` がまだ存在しないときに走る ── ので、`inject` は throw せず `pending` マップに
  退避し、`renderToString` が document 設置後に再生(`flushStyles`)する。各タグの `data-k`
  ハッシュにより、クライアントはハイドレーション時にサーバ送出の `<style>` と重複排除し、
  二重注入しない。SSR でも SSG でも共通 ── どちらもモジュール評価時に実ブラウザの `<head>`
  を持たない。
- **ハイドレーションは SSR と SSG で完全に同一。** クライアント経路はマークアップがリクエスト
  由来かビルド由来かを気にしない。SSG はハイドレーションのコードを増やさない。
- **SSG 固有の論点**(SSR プリミティブが乗るまで後回し):*どの*パスを prerender するかの列挙
  (ルーターからのルート一覧、または明示配列。動的パラメータは `getStaticPaths` 相当の列挙子が
  欲しい)と、リクエスト時サーバが居ないのでビルド時に*データ*を焼き込むこと ── ここで
  Async/Suspense(`resource`)と直列化スナップショットが効く。完全静的(イベントのみ・フェッチ
  無し)ならどちらも不要。既存の hash ルーター / GitHub Pages 路線は、各ルートを `index.html`
  に prerender する話と綺麗に合成できる。

### `kanabun generate` ── SSG コマンド

上で約束した prerender ループは **`kanabun generate [entry]`**
(`packages/cli/src/generate.ts`)として実装した:2 つのコアプリミティブを静的ファイルに
変える Bun 層。SSG の **config** モジュールを import し、各ルートで
`renderToString(() => render(path))` を呼び、マークアップを HTML ドキュメントで包んで
`<outdir>/<route>/index.html` に書き出す(`/` → `index.html`、`/about/` →
`about/index.html`)。

- **config が SSG の契約。** `{ routes?, render(path), client?, title?, document? }`。
  `render` はパスごとのビューを返す。`routes` は既定 `["/"]`。`client`(config ファイル
  からの相対で解決)は `Bun.build` で **一度だけ** バンドルされ全ページから参照される ──
  これで静的 HTML がハイドレートする。`document` を渡せば組み込みのシェルを上書きできる。
  `client` 無しの config は静的のみ(クライアント JS を出さない)。
- **新しい描画経路は無い ── 純粋なオーケストレーション。** `generate` は
  `renderToString`(コア)+ `Bun.build` + `fs` だけで、上で示した「薄い CLI の prerender
  ループ」そのもの。描画はコアでランタイム非依存のまま、ファイル書き出しとクライアント
  バンドルだけが CLI に居る。
- **`build` 同様 never-throw。** 不正なエントリ・クライアントバンドル失敗(`Bun.build` は
  `AggregateError` を投げ、`build` と同じく `errorMessages` で展開)・`render` の throw は
  すべて `{ success: false, logs }` として報告する。
- **ルート列挙は今のところ明示(`routes` 配列)。** `@kanabun/router` からの一覧取得や
  動的パラメータ向けの `getStaticPaths` 相当の列挙子、ビルド時のデータ焼き込み
  (`resource` の直列化スナップショット)は引き続き follow-up。`examples/ssg` が動く例
  ── 2 ルート・スコープド CSS・ハイドレートするカウンター。
- **`base` パスでサブパス配信に対応。** プロジェクトサイト(例: GitHub Pages の
  `/repo/`)はアセットをプレフィックス配下から配信するので、絶対パスの `/main.js` は 404
  になる。`base`(config か `--base` フラグ。フラグが優先)は先頭・末尾スラッシュ1個に
  正規化してクライアント `<script>` の src に前置する。`DocumentContext` にも公開するので
  カスタム `document` が正しいアセット/リンク URL を組める。アプリ内リンクを base 相対に
  保つのはアプリ側(将来の router 相対 `<Link>`)の責務で、`generate` の仕事ではない。

これも同じ基準 ── 依存ゼロ・`packages/core` はランタイム非依存・全ソース カバレッジ 100%・
`tsc` クリーン・ドキュメントはバイリンガル。

## 非同期 / Suspense(Phase 6)

`resource` は非同期関数をリアクティブな状態に変え、`<Suspense>` はそのロード中に
fallback を表示します。Phase 6 の他と同様、どちらも既存の仕組み(signals + owner
ツリーの context)に乗るだけで、コアに新しいプリミティブを足しません。

- **resource は 3 つの signal + version カウンタ。** `value`・`loading`・`error` は
  素の signal。resource を読む(`data()`)と `value` を読み、`data.loading()` /
  `data.error()` は残り 2 つのアクセサ ── プロパティゲッターの魔法は使わず明示 getter
  (フレームワークの規約)。load ごとに単調増加の `version` を捕捉し、fetch が決着しても
  version が最新でなければ破棄する。この 1 つのチェックが resource を**レース安全**に
  する:遅い先行リクエストは、より速い新しいリクエスト(や `mutate`)が既に書いた値を
  上書きできない。
- **source はリアクティブで、未準備なら待機。** `resource(source, fetcher)` は fetch を
  `source()` を読む `effect` で包むので、source が変われば再取得する。source が
  `false`/`null`/`undefined` は「未準備」を意味し、進行中の load をキャンセル
  (`version` を進める)してアイドルする(Solid と同じ)。source 無しの
  `resource(fetcher)` は一度だけロードする(source は定数 `true`)。
- **fetcher 呼び出しは 1 microtask 遅延。** `load` は
  `Promise.resolve().then(() => fetcher(…))` とするので、**同期**の throw も rejection に
  なり(エラー処理を統一)、解決前に `loading` が確実に `true` に観測される ── 値を同期で
  返す fetcher でも同様。
- **エラーは owner ツリーではなく `error()` で公開。** 拒否された fetch はリアクティブ
  グラフに throw せず `error` を立てる。最寄りの `<ErrorBoundary>` へ転送するには
  リアクティブな*読み取り*中に throw する必要があり、eager・ボトムアップのランタイムでは
  正しい境界へ綺麗に結び付けられない ── ハイドレーションを形作るのと同じ「カーソル無し」の
  制約。`error()` の公開は予測可能で UI に選択を委ねられる;resource 専用の
  `<ErrorBoundary>` 連携はコンパイラ/マーカーと共に将来。
- **`<Suspense>` は子を一度だけ生成し、あとは*選ぶ*だけ ── `<ErrorBoundary>` と同じ。**
  `SuspenseContext`(`pending` カウンタを増減する小さなレジストリ)を提供し、子を**その
  context の下・専用 `createRoot` で一度だけ**生成する。サブツリーで作られた resource は
  `useContext` でレジストリを見つけ、ロード中はそれを increment するので、描画 thunk は
  `pending()` を読んで fallback か(既に生成済み・生かしてある)子かを返すだけ。一度だけ
  生成するのが要:`pending` が 0 になってから子を遅延生成すると、隠すたびに resource が
  破棄→decrement→再表示→resource 再生成…と無限ループする。隠れている間も生かすのは要素子を
  持つ `<Show>` と同じ。
- **初回ロードのみサスペンド。** resource は初回成功(`resolvedOnce`)まで境界に登録し、
  以降の `refetch()` は `loading` を立てるが再 increment しないので、直前の値が画面に残る
  (インラインのスピナーには `loading()` を読む)。これは「再検証中は古い内容を見せる」
  一般的な挙動で、これも Solid と同じ。
- **子は関数(遅延)── `<Show>`/context/`<ErrorBoundary>` と同じ。** thunk で包めば
  resource が境界の*下で生成*され、context を見られる;素の eager な子は `<Suspense>` の
  実行前に作られ、何にも登録されない。
- **`lazy()` は `<Suspense>` の「欠けた相棒」。** `lazy(() => import("./X"))` は動的
  `import()` の背後にコンポーネントを遅延配置し、バンドラがそこでコード分割する。返された
  コンポーネントは(resource と同じ増減レジストリで)最寄りの `<Suspense>` に**初回ロード中**
  だけ登録し、解決後はモジュールの `default` エクスポートをその場に描画する。import の
  Promise は**ファクトリ側にキャッシュ**されるので、複数インスタンス ── あるいは事前の
  `Lazy.preload()` ── が 1 本のネットワーク要求を共有する。モジュール解決*後*に生成された
  インスタンスは同期的に描画される(fallback のフレームを挟まない)。**import 失敗時**は
  error をリアクティブグラフへ surface し、外側の `<ErrorBoundary>` が捕捉できる ── ここに
  一つ機微がある: コンポーネントは view を**自分が所有する** `computed`(=境界の下)で構築
  する。呼び出し側に返した素のリアクティブ thunk は*外側*の render エフェクトで再挿入される
  ため、そこからの throw は境界を抜けてしまう;所有された `computed` は throw を正しい owner に
  留める。import 解決前に破棄された場合は静かにキャンセルする(境界を release し、描画も
  throw もしない)。ここに `Bun.*`/バンドラ依存は無い ── コード分割は `import()` 地点での
  バンドラの仕事で、core はロード状態を編成するだけ。

これも同じ基準 ── 依存ゼロ・`packages/core` はランタイム非依存・全ソース カバレッジ 100%・
`tsc` クリーン・ドキュメントはバイリンガル。

## CSS HMR(Phase 6)

dev サーバーは従来、どのファイル変更にも **全リロード** で応じていました。最も頻度の高い
内側ループの編集 ── スタイル調整 ── でも、まさにそのスタイルを当てている対象のアプリ状態
(開いたメニュー、入力途中のフォーム、深いルート)を毎回捨てていたわけです。CSS
ホットリプレースは、設計が他で引く一線を越えずに、このケースだけを正します。

- **なぜ CSS だけで、コンポーネント HMR ではないのか。** 真の状態保持 HMR は、変更された
  *モジュール* を再評価し、状態を保ったままライブツリーへ差し込みます。これには **何を・
  どこに** 差し替えるかを知るための **コンポーネント境界と描画マーカー** が必要で、それは
  まさにコンパイラ/変換が提供するもの(React Fast Refresh、Vite)です。kanabun は
  **コンパイラ無し**・**VDOM 無し** のランタイム JSX で、描画は eager かつ bottom-up なので、
  差し替える境界が存在しません(ノード単位のハイドレーション採用が対象外なのと同じ理由 ──
  [SSR](#ssrハイドレーションssgphase-6) 参照)。一方 `.css` ファイルはそのどれにも縛られず、
  ブラウザが再フェッチできる `<link>` にすぎません。よって CSS HMR は、ここで *実際に到達可能な*
  「状態保持 HMR」の切り口であり、最高頻度の編集をカバーします。
- **仕組み。** ファイル監視は各変更を純粋な `changeMessage(filename)` ヘルパーで分類します
  ── `.css` パスは `css:/<path>`(OS のセパレータを URL 形式に正規化)に、それ以外は
  `reload` に。注入されたクライアントランタイムは `css:` メッセージで
  `link[rel="stylesheet"]` を走査し、URL の **pathname** が変更パスに一致する各 link を、
  キャッシュバスティングクエリ付きで複製し、複製のロード完了後に旧 link を外します(ちらつき無し)。
  何も再実行しないのでアプリ状態は無傷です。**フォールバック:** 一致するスタイルシートが
  無ければ(例: `.css` が `<link>` ではなく JS モジュール経由で import されている場合)全リロードに
  落ちるので、編集が黙って捨てられることはありません。
- **なぜ純関数か。** 判定を監視コールバックから `changeMessage` に切り出すことで、ソケットや
  実ファイルシステムイベントを立てずにユニットテストできます(ソケット経路は `.css` を書いて
  `css:` メッセージ到達を確認するライブテストでカバー)。`createDevHandler` をサーバーから
  切り出しているのと同じ流儀です。
- **スコープド `css\`…\`` は意図的にホットスワップしない。** 内容ハッシュでクラス化されるため、
  本体を編集するとクラスが変わり ── モジュール再実行、すなわちコンパイラが要るコンポーネント
  HMR が必要になります。これらの編集は全リロードのまま。状態保持で差し替えられるのは外部
  スタイルシートの部分です。

これも同じ基準 ── 依存ゼロ・CLI が唯一の Bun 依存層・全ソース カバレッジ 100%・
`tsc` クリーン・ドキュメントはバイリンガル。

## アイランド / 部分ハイドレーション(Phase 7) ── 設計メモ

> 状態: **計画段階、未実装。** コードを入れる前に、方針とそれが何故適合するのかを
> 記録しておく。実装が選択を迫ったら、その都度この節を更新すること。

枠組みの判断: **kanabun の部分ハイドレーションは、自動解析でも resumability でもなく、
明示的な「アイランド(islands)」モデルにする。** ページの大半は静的なサーバ描画 HTML で、
*アイランド*として印を付けたコンポーネントだけがクライアントへ送られ実行される。各アイランドは
独立したマウント点であり、その周囲の静的な外殻にはクライアント JS が一切乗らない。

- **なぜページ全体のハイドレーションより、このアーキテクチャにアイランドの方が合うのか。**
  現状の `hydrate` はコンテナをクリアしてツリー全体を再描画する(ノード単位の採用=adoption は
  しない ── eager でボトムアップな JSX がコンパイラ無しでは採用を不可能にする理由は上記「SSR・
  ハイドレーション・SSG」参照)。アイランドはこの制約を*無害化*する: 各アイランドは*小さく独立した*
  コンテナで、同一のサーババイトに対して自分のサブツリーだけをクリア&再描画し(フラッシュなし)、
  どのアイランドにも属さない部分は再描画もされず送信もされない。採用問題を解くのではなく*回避*し、
  しかも**コンパイラ無し**で実現する ── 創設時の制約に忠実。再利用するのは同じ2つのプリミティブ
  (サブツリーへの `renderToString`、コンテナへの `hydrate`)で、アイランドはその合成であり、
  第3の描画経路ではない。
- **境界は明示的・手動 ── 他の機能と同じく必然として。** コンパイラが無い以上、どのコンポーネントが
  インタラクティブかを*検出*する手段はないので、作者が印を付ける: `<Island name="Counter"
  props={…}>…</Island>` という境界(アイランドを推論するフレームワークより、Astro の `client:*`
  ディレクティブに近い)。サーバ側ではアイランド名と props を載せたラッパ要素に直列化される ──
  例 `<div data-island="Counter" data-props='{"start":0}'>…children…</div>` ── 子はその中へ
  通常どおり描画される(初期描画 / SEO は不変)。クライアント側の**レジストリ**が `name → Component`
  を対応付け、クライアントエントリは全ての `[data-island]` を走査し props を復元して、それらだけを
  mount する。他は一切実行されない。
- **props は境界をクロージャでなくデータとして越える。** サーバ→クライアントの隙間は直列化の隙間:
  アイランドの props は JSON 直列化可能でなければならない。クロージャ・signal・DOM ref は越えられない
  ── ライブなデータが要るアイランドはクライアント側で import / 購読するか、素の値を受け取って自分の
  リアクティブ状態を組む。これはバグではなく、声高に文書化すべき本物の制約。
- **各アイランドは独立した root。context と所有権ツリーは越えない。** アイランドは独立に mount される
  ので、サーバ側の祖先が提供した値をアイランドが `useContext` で読むことはできない ── 所有権ツリーは
  直列化境界で破棄されている。アイランド間の共有状態は普通の JS のやり方で表す: アイランドが import する
  モジュールレベルの singleton signal / ストア(所有権ツリーは別でも、モジュールスコープはクライアントで
  共有される)。
- **実際の payload 削減はコアでなく CLI に宿る。** コアはアイランドを*表現*できる(境界 + レジストリ +
  props の (de)serialize)し、それだけで*実行*の面ではハイドレーションは部分的になる。だがアイランドの
  目的は*より少ない JavaScript*を送ることで、それにはビルド層(`packages/cli`、Bun 専用層)が
  **アイランド単位で code-split** し、ページがそこに含まれるアイランドのチャンクだけを取得するように
  する必要がある。その分割が無ければ結局すべてのアイランドコードを送ってしまう。バンドル分割は本質的に
  Bun / バンドラの仕事なので CLI が正しい置き場所であり、コアはランタイム非依存のまま。
- **スコープ境界。** *コア(小さい):* `<Island>` 境界・レジストリ・JSON props の (de)serialize、
  いずれも既存プリミティブの再利用。*CLI:* アイランド単位のチャンク化 + ページ上のアイランドを読み込み
  mount するクライアントブートストラップ。*明確に対象外:* アイランドの自動検出(コンパイラが要る)・
  ノード単位の採用(同上)・resumability(再実行が一切ない ── 本質的にコンパイラ / 直列化の重いモデルで、
  ランタイム JSX 設計と矛盾する)。順序は Phase 6 と同じ: まずコアの境界(動く `examples/ssr` の
  アイランドデモ付き)、次に CLI の分割。

これも同じ基準 ── 依存ゼロ・`packages/core` はランタイム非依存・全ソース カバレッジ 100%・
`tsc` クリーン・ドキュメントはバイリンガル。
