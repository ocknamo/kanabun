# 設計判断

*[English](./decisions.md) | 日本語*

このドキュメントは kanabun を形づくる選択と ── 同じくらい重要な ── 検討して見送った
代替案を記録します。初期のプロダクト/設計インタビューの結論をまとめたものです。

## プロダクトの位置づけ

- **動機:** *実用ツール*。学習だけでなく、実プロジェクトで使えるものを目指す。
- **成功の定義(約1年):** kanabun 上で TodoMVC が動き、他者が試せる程度にドキュメント
  が揃っている状態。
- **最初のドッグフーディング対象:** 設計検証は TodoMVC 風のデモで十分。具体的な実アプリ
  は後で。
- **直近のスコープ:** 横展開の前に、まず**コアを鉄壁にする**(Phase 1)。

## 技術的な中核判断

### 1. 仮想 DOM ではなく signals でリアクティブに

変わった所だけ更新する。仮想 DOM も差分計算も実装しない。コアは DOM なしで完全に
ユニットテストできる、純粋なリアクティブグラフ。

### 2. 明示 getter(Solid 流)── ゆえに**コンパイラなし**

リアクティビティの表記は `const count = signal(0)`、読みは `count()`、書きは
`count.set(v)`。

これは最も影響の大きい判断でした。元の要求にあった次の2つが両立しないからです。

- *Svelte 風の手触り*(`let count = $state(0); count++`)
- *コンパイラなし*

この2つは共存できません。`count++` のような素のローカル変数代入を実行時に検知するのは
不可能で、Svelte 5 の `$state` が魔法に見えるのは**コンパイラ**がそのコードを追跡可能な
`get`/`set` 呼び出しに書き換えているからです。つまり「Svelte の手触り」は本質的に
コンパイラの産物です。

検討した3つの解決策:

| 案 | 手触り | コンパイラ | 結論 |
| --- | --- | --- | --- |
| **A. 明示 getter(Solid)** | `count()` / `count.set()` | 不要 | **採用** |
| B. Proxy オブジェクト(Vue) | `s.count` / `s.count++` | 不要 | 却下: Proxy のエッジケース(分割代入で購読が切れる、ネスト/配列/Map 対応)が「鉄壁」を難しくし、テンプレート境界では結局 getter が要る。 |
| C. Svelte 風(`count++`) | 最良 | **必須** | 当面却下: コアを固める**前に**コンパイラという大物を背負う ── 選んだ優先度の逆。 |

**A** を採用した理由は、他のすべての回答(*コンパイラなし*・*コア優先*・*集中投下*)と
一直線に整合する唯一の案であり、SolidJS が既に実証済みの戦略だからです。`()` は小さく
正直なコストで、ソース上に見える**購読そのもの**です。

これは C を永久に閉ざすものではありません。将来コンパイラが同じランタイムの上に
`$state` 糖衣を載せることは可能です。A は床であって天井ではありません。

### 3. テンプレートは JSX(後のフェーズ)

テンプレートを導入する際は JSX/TSX にします。TypeScript が要素・属性・props・埋め込み式を
ネイティブに型チェックし、エディタの補完やハイライトも無償で効きます。kanabun が用意する
のは JSX 型定義と小さな `jsx`/`jsxs`/`Fragment` ランタイムだけ ── **LSP も独自 DSL も
作りません**。(tsconfig は既に `jsxImportSource: "@kanabun/core"` に配線済み。)

### 4. ツールには Bun、ただしコアはランタイム非依存

`@kanabun/core` は標準 JS / Web API のみを使うので、どこでも動きます。Bun は開発体験の
ため(テストランナー、将来はバンドラ/dev サーバー)だけに使い、コアを汚さない薄い CLI/dev
層に閉じ込めます。

### 5. 依存ゼロ ── ランタイム*かつ*開発環境

「依存ゼロ」は出荷バンドルだけでなく開発環境にも及びます。

- **ランタイム:** ゼロ。コアは何も import しない。
- **開発:** 型専用の依存 `@types/bun` を1つだけ、意図的な例外として許可
  (`bun:test` / Bun の型を提供する)。文字通りのゼロを目指して手書きの ambient シムを
  試作しましたが、`@types/bun` を選びました。新しいテストマッチャーを使うたびに拡張が
  必要なシムより、正確で保守された型のほうが優れます。
- **TypeScript** は本プロジェクトが許可するツール(憲章「Bun と TypeScript のみ」)であり、
  `package.json` にベンダリングせず `bunx tsc` でオンデマンド取得します。
- **CI 基盤**(`actions/checkout`、`oven-sh/setup-bun` などの GitHub Actions)は
  プロジェクトの依存グラフに含まれず、この規則の対象外です。

結果として `bun install` が入れるのは `@types/bun` のみ、ブラウザに届くのは標準 JS のみ。

## 規約とツール

- **テストファイル名**は `*.spec.ts`。
- **パッケージマネージャ/ランナー**は Bun。`bun test`、`bun run typecheck`
  (`bunx tsc --noEmit`)を使い、npm/yarn 相当は使わない。
- **カバレッジ**は Bun 内蔵の `--coverage`(text + lcov)で計測し、`bunfig.toml` に
  閾値 0.9 を設定。現状コアは 100%。
- **CI**(`.github/workflows/ci.yml`)は push / PR ごとに型チェック・テスト・カバレッジを
  実行。
- **依存の追加**は明示的に正当化すべき危険信号。`skeptical-reviewer` エージェント
  (`.claude/agents/`)がタスク完了報告の前にこれをチェックします。

## リアクティブコアの意味論(Phase 1)

伝播アルゴリズムは push–pull の「カラーリング」方式(Reactively/Solid 流)。後から変える
のが最もつらい次の2つの意味論を、ここで意図的に決め切りました。

- **glitch-free:** ダイヤモンド(`a → b, a → c, (b,c) → d`)で `a` への1回の書き込みが
  各ノードを高々1回しか再計算せず、不整合な中間状態(`d` の一時的な誤値)を見せない。
  テストで検証済み。
- **自動購読解除:** 依存は毎回採り直すので、あるソースを読まなくなった計算はそのソースから
  購読解除される(リークなし、動的依存も自然に動く)。テストで検証済み。

破棄はリーフで明示的に行います。`effect` は disposer を返し、`onCleanup` / 返り値の
クリーンアップが後始末を担います。

所有権は(さらに先送りせず)Phase 2 で必要になった時点で追加しました。
`createRoot(fn => …)` がスコープを作り、その中で生成された計算はまとめて破棄され、
再実行する effect は前回の実行で作った子を破棄します(リークなし)。これは Phase 1 の
伝播 ── とそのテスト ── に手を入れない**追加レイヤー**として実装しました。

## 描画の判断(Phase 2)

### ランタイム JSX、コンパイラなし

`jsx`/`jsxs`/`Fragment` は呼ばれた時点で**実 DOM を即時生成**します
(hyperscript + signals 方式)。仮想 DOM も差分計算も独自コンパイラもありません。
これは「ランタイムのみ」の判断を守りつつ、JSX の型チェックとエディタ支援を無償で得ます
(TypeScript は `jsxImportSource` 経由で我々の `JSX` 名前空間から型を解決)。
コンポーネントは**一度だけ**実行され(Solid モデル)、再実行するのはリアクティブな式だけです。

### 反応式の規約:関数がリアクティブ

コンパイラがないので、何がリアクティブかをソースが示す必要があります。規則は
**子・属性の値が関数ならリアクティブ**(`effect` でラップし、コメントマーカーで位置を
固定)、それ以外は一度だけ設定。`on*` は常にイベント。よって `{count}` や
`{() => count() * 2}` はリアクティブ、`{count()}` は一度読むだけ。これがコンパイラ
なし選択のための唯一の儀式で、小さく明示的です。

### DOM 依存なしで DOM をテストする

レンダラは DOM を必要としますが、Bun は DOM を持たず、jsdom / happy-dom は依存ゼロの
方針に反します。そこでレンダラは `globalThis.document` を遅延解決し、テストはリポジトリ内の
小さな DOM モック(`test/dom-mock.ts`)を差し込みます。import 時に `document` を要求しない
ので、コアはどこでも読み込めます。さらに例は CI で `bun build` によりビルドし、実際の
JSX トランスフォームをエンドツーエンドで検証します。

## 制御構文の判断(Phase 3)

### keyed リストを2層で

最難関 ── 配列の効率的な更新 ── を、独立してテスト可能な2つの小さな部品に分けました。

- `mapArray`(`<For>` 内)はアイテムごとに**ノード identity を安定化**します。参照で
  キーづけし、アイテムにつきノードを一度だけ生成、並べ替え/挿入で再利用、各アイテムは
  独自の `createRoot` で動くので、削除されたアイテムのリアクティビティは破棄されます。
- `reconcileNodes`(DOM ランタイム)は identity を基準に DOM を目標ノード列へ同期し、
  消えたものを削除、本当に位置がずれたノードだけを移動します(後ろから前へアンカリング)。

組み合わせることで、全再構築なしの keyed 更新になります。アルゴリズムは LIS 最適な最小移動
(Solid の udomdiff)ではありませんが、O(n) で正しく、TodoMVC には十分で、同じ継ぎ目の裏で
後から差し替えられます。

### `<Show>` / `<For>` は魔法ではなくコンポーネント

これらは一度実行されリアクティブな thunk を返す普通のコンポーネントで、他のリアクティブな子と
**まったく同じ挿入経路**に乗ります(レンダラ側に特別扱いなし)。`<Show>` は条件を boolean に
メモ化するので、真値の間で条件が変わっただけでは子を入れ替えません。

`<Show>` の破棄はコンパイラではなく「関数は遅延」規約に従います。素の要素の子
(`<Show><Child/></Show>`)は一度だけ生成され、非表示の間は**切り離されるだけ**でリアクティビティ
は生存します。一方、関数の子(`<Show>{() => <Child/>}</Show>`)は遅延生成されるので、非表示で
子のスコープが(owner ツリー経由で)破棄され、表示で再生成されます。両挙動ともテストで固定して
あり、トレードオフは偶然ではなく明示的です。

## CLI の判断(Phase 5)

`@kanabun/cli` は**唯一**の Bun 依存層で、コアは Bun/Node API を一切 import しません。
`kanabun` コマンドは3つのサブコマンドを持ちます。

- `build` は `Bun.build({ target: "browser" })` のラッパー(esbuild/Vite なし)。例外を
  投げず、解決不能なエントリやコンパイルエラーは `{ success: false, logs }` として返します。
- `dev` は Bun 内蔵の `Bun.serve` 上の薄いサーバー。HTML エントリを配信(ライブリロード
  スニペットを注入)し、TS/TSX をオンデマンドでバンドルし、ファイル変更時に WebSocket で
  **全リロード**を送ります。状態保持 HMR は(最も重く効果が薄いため)ロードマップの
  「まず全リロード」に従って先送り。リクエストハンドラ(`createDevHandler`)を分離し、
  ソケットを立てずにユニットテストできるようにしています。
- `create` は埋め込みテンプレートから動くプロジェクトを生成(ネットワーク不要)。

CLI もコアと同じ基準(カバレッジ 100%)を満たし、実 dev サーバーのテスト(エフェメラル
ポートで起動 → HTML + バンドル JS を取得 → ファイル変更で WebSocket リロード発火を検証)も
含みます。

## ロードマップ(要約)

- **Phase 0 — 足場:** Bun プロジェクト、ワークスペース分割(`core` と将来の `cli`)、
  `bun test`、CI + カバレッジ。✅
- **Phase 1 — signals コア:** `signal` / `computed` / `effect`、バッチ、クリーンアップ。
  glitch-free + リークなし、完全にユニットテスト済み(カバレッジ 100%)。✅
- **Phase 2 — JSX ランタイム + 描画:** `jsx`/`jsxs`/`Fragment`、`render`、
  `createRoot`。細粒度のリアクティブ DOM、動くカウンター(カバレッジ 100%)。✅
- **Phase 3 — 制御構文とリスト:** `<Show>`、`<For>`(keyed 更新、2層の
  `mapArray` + `reconcileNodes`)。TodoMVC が動く(カバレッジ 100%)。✅
- **Phase 4 — コンポーネントモデルと DX:** リアクティブ props・children・バインド・`ref`
  は Phase 2〜3 で動作済み。`onMount` と `mergeProps` / `splitProps` を追加。
  ⏳ 残り: `context` とスコープド CSS(どちらも設計判断が必要 ── 下記)。
- **Phase 5 — Bun 連携:** `create` / `dev` / `build` CLI、WebSocket での全リロード dev
  サーバー。Bun 専用層、カバレッジ 100%。✅
- **Phase 6 — 堅牢化(任意):** SSR/ハイドレーション、ルーター、状態保持 HMR など。
